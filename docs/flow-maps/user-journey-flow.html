1

Register — create an account

The visitor fills the sign-up form; the server hashes the password, saves the user, and hands back a session.

  1. FE
    frontend/src/pages/Signup.jsx›submit()

    Validates the password rules (length, upper, number, symbol) client-side, then calls the signup API.

  2. FE
    frontend/src/api/authApi.js›signup()
    POST /api/auth/signup

    Sends email, password, name, phone through the shared axios client.

  3. API
    backend/src/routes/auth.routes.js→controllers/auth.controller.js›signup()

    Passes through authLimiter (rate limit) and validate(signupSchema); rejects duplicate emails with a conflict.

  4. DATA
    backend/src/models/User.js›setPassword() · user.save()

    Hashes with bcrypt (cost 12) and writes the user document.

    MongoDB · users
  5. SVC
    backend/src/utils/tokens.js›signAccessToken() + signRefreshToken()

    Via the controller's issueSession(): sets httpOnly cookies vigno_at / vigno_rt and returns { user, token }.

  6. FE
    frontend/src/store/authSlice.js›setCredentials()

    Persists the session to localStorage['vigno.auth'] and advances to the OTP verify screen (VerifyContact).

2

Log in — session & the auth guard

Password is checked, a single active session is enforced, and every later request is verified by middleware.

Signing in
  1. FE
    frontend/src/pages/Login.jsx›submit()→api/authApi.js›login()
    POST /api/auth/login
  2. API
    backend/src/controllers/auth.controller.js›login()

    Loads the user with +passwordHash and checks User.comparePassword() (bcrypt). If 2FA is on → returns { twoFARequired, challenge } and stops (second step: verify2fa()).

  3. DATA
    models/User.js·findByIdAndUpdate($inc tokenVersion)

    Single active session: bumping tokenVersion instantly invalidates any other logged-in session. Then issueSession() re-issues tokens + cookies.

  4. FE
    store/authSlice.js›setCredentials()→navigate()

    Lands on /app (or /app/admin for admins).

On every request after login
  1. FE
    frontend/src/api/axiosClient.js·request interceptor

    Attaches Authorization: Bearer <token> and sends cookies (withCredentials). On a 401 it silently calls POST /auth/refresh once and replays the request.

  2. API
    backend/src/middleware/auth.js›requireAuth()→verifyAccessToken()

    Reads the token (cookie first, then Bearer), verifies the JWT, re-checks tokenVersion, and sets req.user = {'{ id, role, email }'}. requireRole('admin') gates admin routes.

Buy — payment turns into licenses

A course order is created, paid (Razorpay or the dev/free path), verified, and a license is issued for every lesson.

  1. FE
    frontend/src/pages/Home.jsx›handleBuyCourse()→lib/buy.js›purchaseCourse()
  2. FE
    frontend/src/api/paymentsApi.js›createCourseOrder()
    POST /api/payments/order-course
  3. API
    backend/src/controllers/payments.controller.js›createCourseOrder()

    Refuses a course with no published lessons, applies the coupon, and creates a Purchase.

  4. SVC
    backend/src/services/payments.js›createOrder()

    Real Razorpay order, or the built-in isMock() gateway when no keys are set. ₹0 (100%-off) skips the gateway entirely.

  5. FE
    Razorpay checkout (live) — else free/mock →api/paymentsApi.js›verifyCourse()
    POST /api/payments/verify-course
  6. API
    controllers/payments.controller.js›verifyCoursePayment()→services/payments.js›verifyPaymentSignature()
  7. SVC
    backend/src/services/licenseAuthority.js›issueLicense() — once per lesson

    Writes a License per published lesson (matched on courseKey) and marks the Purchase paid.

    MongoDB · licenses + purchases
  8. FE
    frontend/src/api/licenseApi.js›mine()→license.controller.js › mine()
    GET /api/licenses/mine

    The course page re-reads licenses; isEnrolled flips true and the curriculum unlocks — "Start Learning".

Learn — stream a lesson in the browser

Opening a paid lesson checks ownership, mints a short-lived signed media URL, and streams it into a locked-down viewer.

  1. FE
    frontend/src/pages/ContentViewer.jsx→hooks/useContent.js›useContentItem()→api/mockApi.js›fetchContent()
    GET /api/contents/:id
  2. API
    backend/src/controllers/courses.controller.js›getContent()

    Computes accessible = !isPaid || owned. Not owned → { locked:true } (no media). Download lane → { requiresLauncher:true }.

  3. SVC
    backend/src/services/licenseAuthority.js›hasActiveLicense()

    The ownership gate — looks up an active, unexpired license (admins bypass).

  4. SVC
    services/signedUrl.js›buildHlsUrl() / buildStreamUrl() + storage.js › createMediaUrl()

    Mints an HMAC signed-URL token (short-lived, bound to this content) — for video: an HLS master playlist; for pdf/3d: a signed object URL.

  5. API
    routes/files.routes.js→controllers/files.controller.js›streamFile() / streamHlsAsset()
    GET /api/files/:id/stream · /hls/:asset

    verifySignedToken(), HTTP range (206), and a per-user PDF watermark (addPdfWatermark) before bytes go out.

  6. DATA
    S3 / CloudFront (or local disk fallback)

    The actual media objects; CloudFront-signed segments for adaptive HLS.

  7. FE
    frontend/src/components/VideoPlayer.jsx · PdfViewer.jsx · Model3DViewer.jsx

    hls.js / pdfjs / three.js render inside the page with a drifting email watermark; right-click and downloads are blocked.

6

Play a game — the desktop launcher

Games ship as encrypted files. The Electron app signs in, binds the device, downloads the encrypted bundle, unlocks a one-time key, decrypts in memory, and runs the game.

Prepared once — encrypted at rest (admin upload)
  1. API
    backend/src/controllers/admin.controller.js›uploadContentFile()
  2. SVC
    services/contentCrypto.js›deriveContentKey() / newSalt() → storage.js › saveEncryptedBuffer()

    AES-256-GCM encrypts the game; the object is stored opaque with enc = {'{ iv, tag, salt }'}. The key is never stored — it is re-derivable from a server secret + salt.

    S3 · obj_<id>.enc
Launch & sign in
  1. APP
    launcher/main.cjs›createWindow() · preload.cjs

    Electron main process; the preload bridge exposes window.vigno.* with contextIsolation on, nodeIntegration off.

  2. APP
    launcher/renderer.js login → main.cjs › ipcMain('login')
    POST /api/auth/login  →  auth.controller.js › login()

    Same auth as the web; ES256 access token kept in the process's in-memory session. (2FA → ipcMain('verify2fa').)

  3. APP
    launcher/main.cjs › ipcMain('registerDevice') · fingerprint()
    POST /api/devices/register  →  devices.controller.js › registerDevice()

    Registers a hardware fingerprint; the returned deviceId binds this machine to the license.

Library & download the encrypted game
  1. APP
    launcher/renderer.js › showLibrary() → main.cjs › ipcMain('library')
    GET /api/licenses/mine  →  license.controller.js › mine()

    Lists the user's licenses and keeps only the download-lane (game) ones.

  2. APP
    launcher/main.cjs › ipcMain('download') → streamToDisk()
    GET /api/content/:id/download-url  →  files.controller.js › getDownloadUrl()

    License-gated presigned URL to the still-encrypted object; streamed to disk (fallback: GET /:id/download → downloadEncrypted()).

    disk · userData/vigno/downloads/<id>.enc
Unlock — runs on every "Play"
  1. APP
    launcher/main.cjs › play() — refresh license
    POST /api/licenses/:jti/refresh  →  license.controller.js › refresh()

    licenseAuthority.js › reissueToken() mints a fresh ES256 license token (signed by keystore.js, P-256). Online-only, memory-only.

  2. APP
    main.cjs › play() — verify
    POST /api/licenses/verify  →  license.controller.js › verify()

    licenseAuthority.js › verifyToken() checks signature + expiry + DB revocation + device binding.

  3. APP
    main.cjs › play() — get the one-time key
    POST /api/content/:id/key  →  files.controller.js › getDecryptionKey()

    Re-verifies the token, binds the device on first use, then contentCrypto.js › deriveContentKey() returns {'{ key, iv, tag, alg:aes-256-gcm, expiresInSec:300 }'}. Mismatched devices trip flagIfShared() anti-sharing.

  4. APP
    main.cjs › play() — device-bound in-game token
    POST /api/content/:id/game-license  →  files.controller.js › getGameLicense()

    gameLicense.js › signGameToken() issues an RSA-SHA256 token bound to the Windows MachineGuid, written into the game's _Data for the in-game LicenseGuard.

Decrypt & run — nothing sensitive touches the disk
  1. APP
    launcher/main.cjs › decryptToFile()

    AES-256-GCM streaming (createDecipheriv + setAuthTag) turns <id>.enc into __title.zip in a temp dir; the GCM tag rejects any tampering or wrong key.

    disk · os.tmpdir()/vigno-game-* (transient)
  2. APP
    main.cjs › extractZip() → findGameExe() → child_process.spawn()

    yauzl unzips (zip-slip guarded), locates the Unity <name>.exe + <name>_Data, and launches it.

  3. APP
    main.cjs — cleanup on exit / quit / logout

    The decrypted temp dir is wiped when the game closes; keys and license tokens live only in memory. Only the encrypted .enc ever persists.

▣ The security model at a glance

Two token systems

Stream lane uses short-lived HMAC signed-URL tokens (signedUrl.js). Download lane uses ES256 JWT license tokens signed by a P-256 key in keystore.js (licenseAuthority.js).

Encryption

Everything at rest is AES-256-GCM. The per-content key is derived, never stored — deriveContentKey(id, salt) from a server secret + per-file salt — and only released after license + device checks.

Device binding & anti-sharing

A game license binds to a registered deviceId on first use; the in-game token adds the Windows MachineGuid. Reuse on other machines trips flagIfShared() and can flag the license for an admin.

What lives where

On disk: only the encrypted .enc. In memory only: session token, license token, decryption key. Transient: the decrypted game in a temp dir while it runs.
