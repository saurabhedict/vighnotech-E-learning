How an admin runs the platform & the content pipeline

Every step names the exact file, function, and endpoint.

1

Sign in to the admin console

Same login as any user — the role decides what's unlocked, and every admin API is double-guarded on the server.

  1. FE
    frontend/src/pages/Login.jsx›submit() → api/authApi.js›login()
    POST /api/auth/login

    On success an admin routes to /app/admin (a normal user goes to /app).

  2. API
    backend/src/middleware/auth.js›requireAuth() + requireRole('admin')

    Every /api/admin/* route is guarded by this pair — a non-admin token gets 403.

  3. FE
    frontend/src/pages/admin/AdminDashboard.jsx

    A tabbed shell (?tab=) that mounts one panel each: Overview, Manage Courses → CmsManager, Users → UsersPanel, Reports → ReportsPanel, Commerce → CommercePanel, Licenses → LicensesPanel, Notifications → NotificationsPanel, Site Settings → SettingsPanel, Audit → AuditLog.

  4. API
    Overview → adminApi.stats() → controllers/admin.controller.js›stats()
    GET /api/admin/stats

    Cached headline counts — users, courses, revenue, active licenses.

2

Build the curriculum — the CMS tree

Courses are a nested tree: course → subject → module → chapter → content. Each node inherits the course key from its parent.

  1. FE
    pages/admin/CmsManager.jsx›CourseList.handleAddCourse()
    POST /api/admin/nodes  ·  adminApi.createNode({ kind:'course' })
  2. API
    controllers/admin.controller.js›createNode()

    A course gets courseKey = slug; every child (subject/module/chapter) inherits its parent's key via resolveCourseKey(). This key is what later ties licenses to a course.

    MongoDB · treenodes
  3. FE
    CmsManager.jsx›CurriculumBuilder.handleAddSubject() · SyllabusNode.handleAddChild()

    SyllabusNode is recursive — it computes the next childKind (subject→module→chapter) and calls adminApi.createNode() down the tree.

  4. API
    reorderNodes() · updateNode() · deleteNode()
    POST /nodes/reorder  ·  PATCH/DELETE /nodes/:id

    Drag-reorder, rename, and recursive delete (a chapter delete removes its content too).

3

Create a lesson — the lane is decided here

Adding content to a chapter picks a type, and the type decides the lane — which is the single fork that sends games down the encrypted path and everything else down the streaming path.

  1. FE
    CmsManager.jsx›LessonsList.handleAddLesson()
    POST /api/admin/content  ·  adminApi.createContent({ chapterId, title, type })

    Type is one of exactly four: pdf, video, 3d, game.

  2. API
    controllers/admin.controller.js›createContent()
    → Content.create({ lane: body.lane || defaultLaneForType(type) }) — storageKey still empty
  3. SVC
    shared/src/constants.js›defaultLaneForType()

    The one fork: type === 'game' ? 'download' : 'stream'. Only games are encrypted; pdf / video / 3d all stream.

    MongoDB · contents (lane set)
4

Upload the file — the pipeline branches

The primary path streams the file straight to S3 from the browser, then finalizes. Finalize is where stream and download lanes split.

Primary path · direct browser → S3 (when S3 is configured)
  1. FE
    CmsManager “Upload file” → lib/uploadManager.js › startUpload() → api/adminApi.js › uploadContentFile()
  2. API
    adminApi step 1 → controllers/admin.controller.js›getContentUploadUrl() → storage.js › createDirectUpload() → s3.js › presignPutUrl()
    POST /api/admin/content/:id/upload-url

    Returns a one-hour presigned PUT URL + the object key obj_<id>. (If S3 is off → { supported:false } and the client uses the fallback below.)

  3. DATA
    axios.put(presignedUrl, file) — straight to S3, no backend in the middle

    Bytes never touch our server → no size cap, true end-to-end progress. The raw object lands in S3 identically for both lanes.

    S3 · obj_<id> (raw)
  4. API
    adminApi step 3 → admin.controller.js › completeContentUpload() → storage.js › statObject()
    POST /api/admin/content/:id/upload-complete

    Confirms the PUT landed, then branches on content.lane ↓

STREAM pdf · video · 3d — store as-is
  1. API
    completeContentUpload() → set storageKey, enc.encrypted = false → maybeTranscodeVideo()

    No encryption. The object is served later through a short-lived signed URL. Then the video sub-branch runs ↓

    MongoDB · content.storageKey = obj_<id>
STREAM · video only adaptive HLS transcode
  1. SVC
    admin.controller.js › maybeTranscodeVideo() → services/mediaconvert.js › submitHlsJob()

    Only when type === 'video' && mediaConvertEnabled(). Submits a 3-rung 720/480/360 H.264/AAC ladder; sets hls.status = 'processing'. If it can't submit, the upload still succeeds (progressive-MP4 fallback).

    S3 · hls/<id>/master.m3u8
  2. JOB
    services/transcodePoller.js › pollPendingTranscodes() — every 15s (index.js › startTranscodePoller)

    Polls AWS via mediaconvert.js › getJobStatus() and flips hls.status to ready or failed. Runs on a background interval, not the request.

DOWNLOAD game — encrypt in the background
  1. API
    completeContentUpload() → enc.status = 'encrypting', enc.rawKey = uploadKey → startDownloadEncryption()

    Returns to the admin immediately with { encrypting:true } — the old game stays playable until the new one is ready.

  2. JOB
    admin.controller.js › startDownloadEncryption() → contentCrypto.js › newSalt() + deriveContentKey() → storage.js › saveEncryptedFromObject()

    Streams the raw S3 object → AES-256-GCM → a new .enc object (never buffers a multi-GB game in RAM). Stores enc = {'{ iv, tag, salt, status:ready }'} and frees the raw key. A boot-time recoverEncryptions() resumes any job interrupted by a restart.

    S3 · obj_<id>.enc (encrypted)
Fallback · server-through (S3 off, or smaller files ≤ 200 MB)
  1. API
    adminApi.uploadContentFile() fallback → admin.controller.js › uploadContentFile() (multer memory)
    POST /api/admin/content/:id/upload

    download → saveEncryptedBuffer() (whole-buffer AES-256-GCM); stream → saveBuffer() then maybeTranscodeVideo(). Same outcomes, routed through the server.

  2. FE
    LessonDetailsEditor.handleUploadFile() → adminApi.uploadContentThumbnail()
    POST /api/admin/content/:id/upload-thumbnail

    Thumbnails always take the plain path — storage.js › saveBuffer() + createMediaUrl(), never encrypted, never transcoded. (Course art: uploadCourseThumbnail → /admin/nodes/:id/upload-thumbnail.)

▣ Pipeline principles worth remembering

One fork, four outcomes

defaultLaneForType() is the only branch: game → download (encrypted), and pdf / video / 3d → stream. Everything downstream follows from that single decision.

Keys are derived, never stored

deriveContentKey(id, salt) re-computes the AES-256 key from a server secret + per-file salt. Only salt / iv / tag are persisted (all select:false); the key exists only for the moment of encrypt or decrypt.

Heavy work is non-blocking

Uploads return immediately. Game encryption (startDownloadEncryption) streams S3→GCM→S3; video transcoding is polled every 15s. Both survive a restart via recoverEncryptions / the transcode poller.

Storage picks itself

createMediaUrl() chooses the read URL: CloudFront-signed → presigned S3 → local disk. The same code runs whether you're on a laptop or full AWS.
